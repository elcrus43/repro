# ADR-001 — Исправление проблем с сохранением данных

## Метаданные
- **Дата**: 2026-04-14
- **Статус**: ✅ Принято и исправлено
- **Контекст**: Клиенты, показы и другие данные не сохранялись в Supabase

## Проблемы

### 1. **Клиенты (Clients) — телефоны не сохранялись**
**Проблема**: 
- Форма отправляет `phones` как массив `[phone1, phone2, ...]`
- В схеме БД таблицы `clients` **НЕТ** колонки `phones`
- Есть только `phone` (одиночный TEXT) и нужно добавить `phone_2`
- `supabaseSync.js` не преобразовывал массив `phones` → `phone` + `phone_2`

**Результат**: При сохранении клиента Supabase возвращал ошибку схемы (колонка не найдена), данные **НЕ сохранялись**.

**Решение**:
1. Создана миграция `030_fix_clients_and_showings_save_issues.sql`:
   - Добавлена колонка `phone_2 TEXT`
   - Убедиться что `client_types`, `additional_contacts`, `passport_details` имеют правильные defaults
2. Исправлен `supabaseSync.js`:
   - `ADD_CLIENT`: извлекает `phone` из `phones[0]`, `phone_2` из `phones[1]`
   - `UPDATE_CLIENT`: аналогичная обработка
   - Удаляется поле `phones` перед отправкой в БД

### 2. **Показы (Showings) — отсутствует realtor_id**
**Проблема**:
- При создании показа `realtor_id` мог не быть установлен
- RLS политика требует `realtor_id = auth.uid()` для вставки
- Без `realtor_id` показ **НЕ сохранялся** (ошибка RLS)

**Решение**:
- В `useDbDispatch.js`:
  - `ADD_SHOWING`: автоматически устанавливает `realtor_id` из `currentUser`
  - `UPDATE_SHOWING`: сохраняет/устанавливает `realtor_id`

### 3. **Сделки (Deals) — сохранение работает**
**Статус**: ✅ Проблем не обнаружено
- `deal_date`, `deposit_date`, `deposit_amount` правильно передаются
- Миграция `029` добавила все необходимые поля

## Последствия
- Данные теперь сохраняются корректно
- Миграция `030` должна быть выполнена в Supabase Dashboard
- Обратная совместимость: старые клиенты с одним телефоном продолжат работать
